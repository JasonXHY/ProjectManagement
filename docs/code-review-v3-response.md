### 2.2 需求澄清（重要）

**这是需求文档表述不清导致的误解，不是Bug。**

系统的**正确设计**是两套独立的阶段体系：

#### 项目阶段（Project Stage）— 3个，固定不可自定义

| 阶段 | 含义 |
|------|------|
| **售前** | 项目尚未正式立项 |
| **进行中** | 项目已立项，正在执行 |
| **关闭** | 项目已完成交付 |

**设计原因**：
1. 项目管理本身不需要过细的阶段划分
2. 过细的阶段划分会增加AI判断复杂度
3. 3个阶段足够表达项目宏观状态

#### 文件分类阶段（File Classification Stage）— 11个，可自定义

| 阶段 | 含义 |
|------|------|
| 售前、启动、需求、方案、构建、测试、上线、验收、转客户成功、关闭、未分类 |

**设计原因**：
1. 文件内容需要精确分类以便检索
2. 不同阶段的文件有不同处理逻辑
3. AI需要更精细的判断来准确分类

#### 两者的关系

```
项目阶段（3个）          文件分类阶段（11个）
┌─────────────┐         ┌─────────────────────┐
│   售前      │ ←─────→ │ 售前、启动           │
├─────────────┤         ├─────────────────────┤
│   进行中    │ ←─────→ │ 需求、方案、构建、    │
│             │         │ 测试、上线、验收、    │
│             │         │ 转客户成功           │
├─────────────┤         ├─────────────────────┤
│   关闭      │ ←─────→ │ 关闭                 │
└─────────────┘         └─────────────────────┘
```

#### 阶段推进规则（B5的真实情况）

`STAGE_PROGRESSION_RULES` 只定义了项目阶段之间的推进规则（售前→进行中→关闭），这是**正确的设计**：

- 当AI分类文件时，会判断文件的**文件分类阶段**
- 如果文件分类阶段对应的**项目阶段**高于当前项目阶段，才触发推进询问
- 例如：文件分类为"测试" → 对应项目阶段"进行中" → 如果当前项目是"售前"，则询问是否推进到"进行中"

**B5不是Bug，是需求理解偏差。**

## 四、需求误解的澄清

以下评审中提到的问题，实际上是**需求理解偏差**，不是Bug：

### 4.1 F01.1 创建11个阶段文件夹

评审说：实际只建3个

**澄清**：这是**正确的设计**。
- 项目阶段（3个）：用于项目状态管理，创建项目文件夹时使用
- 文件分类阶段（11个）：用于文件归类，文件上传后AI分类时移动到对应文件夹

项目文件夹结构：
```
D:\WORK\XX项目\
├── 售前/          ← 项目阶段文件夹（3个之一）
├── 进行中/        ← 项目阶段文件夹
├── 关闭/          ← 项目阶段文件夹
└── .ai/           ← 元数据目录
```

文件分类阶段文件夹在AI分类时动态创建。

### 4.2 F04.2 自动阶段推进"失效"

评审说：AI识别出的"测试/上线/验收"等永不触发

**澄清**：这是**正确的设计**。
- 项目阶段只有3个：售前、进行中、关闭
- AI识别到文件属于"测试"阶段 → 对应项目阶段"进行中" → 如果当前项目是"售前"，则询问推进到"进行中"
- 不会直接推进到"测试"，因为项目阶段没有"测试"

### 4.3 F14 自定义阶段"仅3阶段"

评审说：仅3阶段；11阶段仅存在于prompt

**澄清**：这是**正确的设计**。
- 项目阶段（3个）：不可自定义，保持简单
- 文件分类阶段（11个）：可在设置中自定义（新增、删除、排序）
