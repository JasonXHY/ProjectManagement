## 一、项目概述

### 1.1 项目背景

本项目旨在开发一个轻量级、流畅的AI项目管理工具，适用于Windows平台，主要面向项目经理和个人用户。项目支持后期打包分享。

### 1.2 核心需求

- Windows上运行轻量级、流畅的项目管理软件
- 项目页面显示项目阶段（蓝图、启动、售前、验收、项目中止、结束等）
- 对接云端大模型API（用户自行购买，如小米Mimo平台）
- 拖拽文件到对应项目，调用云端模型智能分类文件
- 每个项目拥有独立对话框，通过自然语言与模型交互
- 具备数据分析能力（通过云端模型）
- 每个项目采用类似Agent架构的文件夹，内部存储Markdown文件记录关键信息

### 1.3 设计原则

- **本地优先 + 按需调用大模型**：所有基础数据存本地，只在真正需要AI理解或生成时才调用云端API
- **Token极致优化**：通过RAG、分层记忆、Prompt精简等策略，将每次调用token控制在300～1500之间
- **轻量级运行**：内存占用控制在30～50 MB，启动<0.5秒
- **数据一致性优先**：每次实时检索，确保返回最新内容，避免缓存导致的信息不一致

## 二、技术选型

### 2.1 桌面应用框架

**选择：Tauri 2.0 + React + TypeScript（简化Rust方案）**

- **理由**：
  - 安装包小（3-5MB）
  - 内存占用低（30-50MB）
  - 启动速度快（<0.5秒）
  - TypeScript为主，Rust只处理必要的系统级操作

- **Rust使用范围**：
  - 文件系统操作
  - SQLite数据库操作
  - 系统API调用

- **TypeScript使用范围**：
  - 业务逻辑
  - UI组件
  - API调用
  - 状态管理

### 2.2 UI组件库

**选择：Ant Design X（对话组件） + shadcn/ui（轻量组件）**

- **理由**：
  - Ant Design X提供专业的对话UI组件
  - shadcn/ui提供轻量美观的中后台组件
  - 符合简约现代的设计风格

### 2.3 本地数据库

**选择：SQLite**

- **理由**：
  - 无服务，单文件，极轻量
  - 适合本地存储项目元数据、任务、对话历史

### 2.4 大模型API

**选择：小米Mimo平台（OpenAI兼容协议）**

- **理由**：
  - 用户已有账号和额度
  - 支持OpenAI兼容协议
  - 适合长期使用

### 2.5 向量数据库

**选择：轻量级向量库（如Qdrant Edge或自研方案）**

- **理由**：
  - 嵌入式，无需独立服务
  - 支持本地高效检索
  - 适合文档向量化和语义搜索

## 四、数据模型设计

### 4.1 核心表结构

#### 项目表 (projects)

```sql
CREATE TABLE projects (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL,
    description TEXT,
    stage TEXT DEFAULT 'blueprint',  -- ⚠️ 待定：后期专题讨论确定最终阶段定义
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    status TEXT DEFAULT 'active'  -- ⚠️ 待定：后期专题讨论确定最终状态定义
);
```

#### 任务表 (tasks)

```sql
CREATE TABLE tasks (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id INTEGER REFERENCES projects(id),
    name TEXT NOT NULL,
    start_date DATE,
    end_date DATE,
    progress INTEGER DEFAULT 0,
    status TEXT DEFAULT 'pending',
    dependency_ids TEXT  -- JSON数组，存储依赖任务ID
);
```

#### 文件表 (files)

```sql
CREATE TABLE files (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id INTEGER REFERENCES projects(id),
    name TEXT NOT NULL,
    path TEXT NOT NULL,
    category TEXT,  -- 文件分类：需求文档、技术方案、会议纪要等
    tags TEXT,  -- JSON数组，存储标签
    version INTEGER DEFAULT 1,
    content_hash TEXT,  -- 文件内容哈希，用于版本控制
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

#### 对话表 (conversations)

```sql
CREATE TABLE conversations (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id INTEGER REFERENCES projects(id),
    role TEXT NOT NULL,  -- user 或 assistant
    content TEXT NOT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    token_count INTEGER  -- 本次对话消耗的token数量
);
```

### 4.2 关系说明

- **projects → tasks**：一个项目包含多个任务
- **projects → files**：一个项目包含多个文件
- **projects → conversations**：一个项目包含多条对话记录
- **files**：通过category字段实现文件分类
- **conversations**：记录用户和AI的交互历史

### 4.3 扩展性考虑

- 后期可添加：里程碑表(milestones)、依赖关系表(dependencies)
- 文件标签系统：可将tags字段改为独立的标签表
- 向量索引：可单独存储，不放在主数据库中

## 五、核心功能设计

### 5.1 项目管理功能

#### 5.1.1 项目列表页

- **布局**：表格布局，项目名称、描述、状态放在一行
- **搜索**：支持按项目名称或描述搜索
- **筛选**：
  - 默认隐藏已结束/中止的项目
  - 可切换显示全部项目或仅显示特定状态
  - 状态选项：蓝图阶段、启动阶段、售前阶段、项目中、验收阶段、已结束、已中止
- **排序**：
  - 按更新时间排序（默认）
  - 按创建时间排序
  - 按项目名称排序
- **分页**：每页显示固定数量，支持翻页

#### 5.1.2 项目阶段

⚠️ **待定**：以下项目阶段定义为初始版本，后期需专题讨论确定最终版本

预设项目阶段（暂定）：
1. **蓝图阶段**：项目规划和设计
2. **启动阶段**：项目启动和准备
3. **售前阶段**：售前支持和方案
4. **项目中**：项目执行阶段
5. **验收阶段**：项目验收和交付
6. **项目中止**：项目暂停或终止
7. **结束**：项目完成

> 注意：阶段名称、数量、定义等都可能在后期专题讨论中调整

### 5.2 文件管理功能

#### 5.2.1 文件管理页布局

- **左侧**：项目阶段列表（蓝图、启动、售前、进行中、验收、中止、结束）
- **右侧**：对应阶段的文档列表
- **文档显示**：最新版本或整合文档

#### 5.2.2 文件操作按钮

- **👁️ 预览**：快速查看文件内容
- **✏️ 编辑**：打开文件进行编辑
- **📁 移动**：手动调整文件分类
- **🏷️ 分类**：手动设置文件类别
- **📋 版本历史**：查看文件历史版本
- **📁 打开文件夹**：在系统文件管理器中打开

#### 5.2.3 智能分类系统

- **预设分类**：需求文档、技术方案、会议纪要、测试报告、用户手册
- **自动标签**：根据内容生成关键词标签
- **分类统计**：显示各类别文件数量和占比

#### 5.2.4 版本控制系统

- **版本记录**：每次修改自动保存版本
- **变更比较**：查看两个版本之间的差异
- **版本回退**：恢复到任意历史版本

#### 5.2.5 内容追踪系统

- **进度更新**：自动识别并更新项目进度
- **问题发现**：识别文件中的问题和风险
- **方案新增**：提取文件中的新方案和建议
- **追踪记录**：生成变更日志和影响分析

#### 5.2.6 文件生成系统

- **进度报告**：生成项目进度汇总报告
- **问题清单**：提取所有问题和解决方案
- **方案汇总**：汇总所有技术方案和决策
- **关键信息提取**：为其他agent准备结构化上下文

#### 5.2.7 文件处理流程

```
用户拖拽上传 → 内容提取（本地） → AI分析（云端） → 自动归类 → 更新追踪 → 建立索引
```

### 5.3 对话功能

#### 5.3.1 对话窗口设计

- **基础对话**：支持自然语言问答
- **文档上传**：支持拖拽上传文件，根据文件内容进行对话
- **历史记录**：保存所有对话历史，支持搜索和导出

#### 5.3.2 上下文管理

- **对话历史**：本地保存所有记录，API只发送最近5轮对话
- **RAG检索**：每次检索Top-3相关片段，片段长度限制300字
- **长度控制**：字符数估算，固定格式模板
- **实时检索**：确保返回最新版本的内容

#### 5.3.3 对话处理流程

```
用户输入 → 检查缓存（精确匹配） → RAG检索（Top-3片段） → 组装Prompt → 调用API → 保存记录
```

#### 5.3.4 Token消耗预估

- 系统提示：约500字 ≈ 1000 tokens
- 项目上下文：最多1000字 ≈ 2000 tokens
- 对话历史：最近5轮 ≈ 1000 tokens
- 用户问题：约100字 ≈ 200 tokens
- **总计**：约4200 tokens/次（在主流模型限制内）

#### 5.4.3 页面设计

1. **项目列表页**：表格布局，支持搜索、筛选、排序
2. **文件管理页**：左侧项目阶段，右侧对应文档

## 六、性能与可行性评估

### 6.1 性能目标

| 指标 | 目标值 |
|------|--------|
| 安装包体积 | < 10 MB |
| 冷启动内存 | 30-50 MB |
| 空闲 CPU | < 1% |
| 向量检索（1000文档） | < 200 ms |
| 对话响应（调用API） | 取决于网络 + 模型延迟 |

### 6.2 风险分析与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 大量文档首次索引 | CPU 短暂跑满 | 后台静默索引 + 按需索引 |
| API 调用延迟高 | 对话窗口等待时间过长 | 增加超时重试机制；支持离线模式 |
| 本地向量库损坏 | 检索失效 | 定期备份向量索引；提供重建索引按钮 |
| Windows 环境兼容性 | 部分老系统缺少 WebView2 | 安装包检测并提供下载链接 |

### 6.3 Token成本预估

- 每次对话平均消耗 300～1500 Tokens → 0.0006～0.003 元/次
- 日均 50 次调用 → 0.03～0.15 元/天 → 每月不足 5 元
