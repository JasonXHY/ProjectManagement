# PMAer v0.2.0 版本规划（终稿）

> 版本：0.2.0 | 前置版本：v0.1.2
> 状态：已确认 | 目标：详情页卡片增强 + 项目转交 + Markdown 预览

---

## 一、版本定位

v0.2.0 是 PMAer 的**体验深化版本**，围绕三个目标：

1. **详情页卡片系统** — 按项目阶段差异化展示关键信息，让项目经理进入详情页即掌握项目全貌
2. **项目转交功能** — 支持整体打包和 AI 辅助选择性转交，让项目知识在团队中流动
3. **Markdown 预览** — 统一渲染能力，修复项目摘要纯文本展示的割裂体验

本版本不包含：暗色模式、快捷键、虚拟滚动、拖拽排序（均推迟至后续版本）。

## 二、功能模块详细设计

### 模块 A：详情页卡片系统

#### A1. 整体架构：统计行 + 侧边栏驱动特色卡片

详情页采用现有的左右分栏布局（左侧 200px 侧边栏 + 右侧内容区），卡片系统分为两层：

```
┌──────────────────────────────────────────────────────┐
│  统计行（通用，始终显示）                              │
│  [文件数量]  [里程碑]  [待处理]  [AI 摘要]             │
├──────────────────────────────────────────────────────┤
│  特色卡片区（跟随侧边栏选中阶段动态切换）              │
│  [卡片1]  [卡片2]  [卡片3]  [可选卡片4]               │
├──────────────────────────────────────────────────────┤
│  文件操作区（现有）                                    │
│  [上传区域]  [文件列表]                                │
└──────────────────────────────────────────────────────┘
```

**设计原则：**

1. **统计行始终显示**，4 列等宽网格，不随阶段变化
2. **特色卡片由侧边栏驱动**，根据侧边栏选中项所属的项目阶段分组（售前/进行中/关闭）动态切换
3. **启动阶段归入进行中**，不再单独一组
4. **不做「更多卡片」自定义按钮** — 卡片数量可控，不需要用户配置
5. **视觉规范参考** `.qoderwork/mockups/04-card-system-v020.html`

#### A2. 统计行（所有阶段通用）

4 列等宽网格，gap 16px：

| 位置 | 卡片 | 内容 | 交互 |
|------|------|------|------|
| 1 | 文件数量 | 项目文件总数 | 无 |
| 2 | 里程碑 | 下一步里程碑名称 + 日期 | 点击弹窗查看完整时间轴 |
| 3 | 待处理 | 未被 AI 分析的文件数 | 无 |
| 4 | AI 摘要 | 「查看摘要」「生成/更新」按钮 | 查看摘要打开 800px Markdown 渲染弹窗 |

**图标配色：** 文件数量(蓝色)、里程碑(琥珀色)、待处理(红色)、AI摘要(紫色)

**里程碑弹窗实现要点：**
- 复用 Antd `Timeline` 组件，竖向布局
- 节点状态：已完成(Indigo实心) → 下一步(Amber实心+脉动光晕) → 未到(灰色空心)
- 关键节点（合同/上线/验收）显示 ★ 图标
- 延期节点（日期已过未完成）显示红色标识
- 数据来源：`projects.milestones` JSON 字段

#### A3. 特色卡片区（侧边栏驱动）

特色卡片根据侧边栏选中项映射到项目阶段分组后动态展示：

**售前分组**（侧边栏选中：售前）→ 3 列网格：

| 卡片 | 内容 | 弹窗 | 数据来源 |
|------|------|------|----------|
| 项目信息 | 项目编号、客户名称、联系人、电话、地址、项目经理 | 无 | `projects.metadata` JSON |
| 合同概览 | 合同文件列表 + 签字状态标签 | 完整签字列表 + 统计 | `files` 表按 category 筛选 |
| 项目评估 | 预估合同金额、成本评估、利润率、人天 | 无 | 待确定数据源 |

**进行中分组**（侧边栏选中：启动/需求/方案/构建/测试/上线/验收/转客户成功）→ 3 列网格：

| 卡片 | 内容 | 弹窗 | 数据来源 |
|------|------|------|----------|
| 需求跟踪 | 前 3 条需求（名称+讨论结论+状态） | 完整需求列表 | 待确定数据模型 |
| 关键问题 | 前 3 个问题（优先级色点+状态标签） | 完整问题清单，支持勾选完成 | `projects.metadata.key_issues` |
| 签字追踪 | 签字进度条 + 待签文件列表 | 完整签字文件列表 + 统计 | `files` 表按 `signature_status` 筛选 |

**关闭分组**（侧边栏选中：关闭）→ 3 列网格：

| 卡片 | 内容 | 弹窗 | 数据来源 |
|------|------|------|----------|
| 拓展商机 | 二期/三期规划、未来优化需求 + 状态标签 | 无 | 待确定数据模型 |
| 交付物清单 | 前 3 项交付物 + 签字状态 | 完整交付物列表 | `files` 表按 category 筛选 |
| 项目总结 | 摘要文字（4 行折叠/展开） | 无 | `files` 表 + `ai_summary` |

**「所有文件」选中时**：根据 `project.current_stage` 显示对应分组的卡片。

**待办事项（后续版本细化）**：
- 项目信息卡片：字段扩展（具体新增字段待讨论）
- 项目评估卡片：内容维度和评估算法（参考成本评估数据源）
- 需求跟踪卡片：数据录入方式、关联文件、时间节点展示粒度
- 拓展商机卡片：数据来源和录入流程

#### A4. 首页卡片简化

保持现有卡片信息，仅做以下调整：

1. **移除里程碑时间轴** — 已有 `ProjectTimeline` 从首页卡片中移除，里程碑信息改由详情页统计行和特色卡片展示
2. **文件统计行简化** — 改为 `12 个文件 · 4 待处理`，去掉"已分类"数字（信息冗余）
3. **项目阶段标签** — 在卡片右上角用彩色标签展示当前阶段（售前=金色、进行中=靛蓝、关闭=灰色）

### 模块 B：项目转交功能

#### B1. 功能概述

两种转交模式，覆盖不同场景：

**整体转交** — 将整个项目的文件、元数据、AI 摘要打包导出，适合项目经理交接、项目归档。

**协作转交** — AI 根据描述推荐相关文件，用户确认后可选择导出精简包，适合邀请同事参与项目特定模块。

两种模式共用同一套导出/导入基础设施，区别仅在于文件选择方式。

#### B2. 转交包格式：.pmaer.zip

**文件格式：** ZIP 压缩包，扩展名 `.pmaer.zip`

**包内结构：**

```
project-name.pmaer.zip/
├── handover.json          ← 转交元数据（必须）
├── project-summary.md     ← AI 项目摘要（如有）
├── key-info.md           ← 关键信息汇总（如有）
├── context.md            ← 转交说明（发送方填写）
└── files/                 ← 项目文件（按分类目录组织）
    ├── 售前/
    │   ├── 报价单/
    │   │   └── XX项目报价.xlsx
    │   └── 合同原件/
    │       └── 合同v2.pdf
    ├── 需求/
    │   └── 需求文档/
    │       └── 需求规格说明书.docx
    └── ...
```

**handover.json 结构：**

```json
{
  "format_version": "1.0",
  "type": "full | selective",
  "project": {
    "name": "项目名称",
    "category_type": "stage",
    "current_stage": "进行中",
    "created_at": "2026-06-01T10:00:00Z",
    "milestones": [{ "date": "2026-06-15", "title": "需求确认", "type": "milestone" }],
    "metadata": { "contract_no": "XX-2026-001", "contact_person": "张三" }
  },
  "files": [
    {
      "filename": "需求规格说明书.docx",
      "relative_path": "files/需求/需求文档/需求规格说明书.docx",
      "category": "需求",
      "subcategory": "需求文档",
      "stage": "进行中",
      "file_type": "application/vnd.openxmlformats-officedocument.wordprocessingml.document",
      "file_size": 102400,
      "content_extracted": "（提取的文本内容）",
      "signature_status": "unsigned",
      "ai_summary": "（AI 摘要）",
      "ai_key_info": { "key_findings": [...] }
    }
  ],
  "handover_note": "（发送方填写的转交说明）",
  "exported_at": "2026-06-22T15:00:00Z",
  "exported_by": "PMAer v0.2.0"
}
```

#### B3. 导出流程

**整体转交流程：**

```
用户点击「转交项目」
    ↓
选择「整体转交」
    ↓
填写转交说明（可选）
    ↓
系统打包所有文件 + 元数据 → .pmaer.zip
    ↓
保存对话框：用户选择保存位置
    ↓
导出完成，提示成功
```

**协作转交流程：**

```
用户点击「转交项目」
    ↓
选择「协作转交」
    ↓
填写同事描述（必填）
    例："张工负责前端开发，需要了解 UI 规范和接口文档"
    ↓
填写转交说明（可选）
    ↓
点击「AI 推荐文件」
    ↓
AI 分析文件列表 + 用户描述 → 返回推荐文件列表
    ↓
┌─────────────────────────────────────┐
│  AI 推荐的文件（已勾选）：            │
│  ☑ UI设计规范.docx — 推荐理由...     │
│  ☑ 接口文档v3.pdf — 推荐理由...      │
│  ☑ 客户界面要求.docx — 推荐理由...   │
│                                     │
│  其他文件（可手动勾选）：              │
│  ☐ 需求规格说明书.docx               │
│  ☐ 项目计划.xlsx                     │
│  ...                                │
└─────────────────────────────────────┘
    ↓
用户确认/调整勾选
    ↓
点击「导出」→ 生成 .pmaer.zip
    ↓
保存对话框 → 导出完成
```

**协作转交的 AI Prompt 设计：**

```
你是 PMAer 项目转交助手。根据新同事的角色描述，从项目文件中选择最相关的文件。

## 新同事描述
{user_description}

## 项目文件列表（含分类信息）
{file_list_with_categories_and_summaries}

## 项目摘要
{project_summary}

请返回 JSON 格式：
{
  "selected_files": [
    {
      "filename": "文件名",
      "reason": "选择理由（一句话）"
    }
  ],
  "key_context": "新同事需要了解的关键上下文概述（2-3 句话）"
}

选择原则：
1. 优先选择与角色描述直接相关的文件
2. 合同、报价等商务文件仅在明确需要时选择
3. 会议纪要选择最近 3 次相关的
4. 不超过 15 个文件
```

**实现要点：**
- AI 推荐调用复用 `aiService.chat()` 接口
- 文件列表传入时包含：文件名、分类、子分类、AI 摘要（前 100 字）
- AI 返回后解析 JSON，在前端渲染为可勾选列表
- AI 推荐失败时 fallback 为空列表，用户手动勾选
- 推荐理由显示在每个文件旁，帮助用户判断

#### B4. 导入流程

```
用户在首页点击「导入转交」按钮
    ↓
选择 .pmaer.zip 文件（文件选择器或拖拽）
    ↓
PMAer 内部解压 ZIP，读取 handover.json
    ↓
校验格式版本号（format_version），不兼容则报错
    ↓
┌─────────────────────────────────────┐
│  导入预览                             │
│                                     │
│  项目名称：XX 管理系统               │
│  文件数量：23 个文件                  │
│  项目阶段：进行中                    │
│                                     │
│  转交说明：                          │
│  "该项目下周需交付，重点关注..."     │
│                                     │
│  [项目摘要预览（前 200 字）]          │
│                                     │
│  导入后项目名称：[XX 管理系统     ]   │
│  （可修改，默认保留原名）             │
│                                     │
│              [取消]  [确认导入]       │
└─────────────────────────────────────┘
    ↓
用户确认导入
    ↓
创建新项目（使用用户填写的名称）
    ↓
解压文件到项目文件夹对应阶段/子分类目录
    ↓
导入 handover.json 中的元数据到 projects 表
    ↓
导入文件记录到 files 表（保留分类信息）
    ↓
导入里程碑数据
    ↓
导入完成，自动进入项目详情页
```

**冲突处理：**
- 如果已存在同名项目，导入对话框提示"项目名称已存在，是否自动重命名为 'XX 管理系统 (导入)'？"
- 用户可选择改名或覆盖（覆盖时创建副本而非删除原项目）

**格式兼容性：**
- `handover.json` 中的 `format_version` 字段用于版本校验
- v0.2.0 只支持 `format_version: "1.0"`
- 未来版本导入旧格式时向后兼容，导入新格式时提示升级

### 模块 C：Markdown 预览优化

#### C1. 问题现状

1. AI 对话回复已用 `react-markdown` 渲染，但样式粗糙
2. `.ai/project-summary.md` 在项目首页弹窗中以纯文本展示，体验割裂
3. 文件列表中的 `.md` 文件无法预览渲染效果
4. 无代码语法高亮

#### C2. 统一 Markdown 渲染组件

创建 `src/components/MarkdownPreview/index.tsx`，基于现有 `react-markdown` + `remark-gfm`，新增以下能力：

| 能力 | 技术方案 | 说明 |
|------|----------|------|
| GFM 表格 | `remark-gfm`（已有） | 需求文档中常见的表格 |
| 代码高亮 | `rehype-highlight`（新增） | highlight.js 样式 |
| HTML 嵌入 | `rehype-raw`（新增） | 支持内联 HTML |
| 样式统一 | 设计规范的 CSS 变量 | 与整体 UI 一致 |

**样式规范：**
- 标题层级：h2 用 18px/semibold，h3 用 16px/medium，h4 用 14px/medium
- 代码块：`#F8F7F4` 背景，`#E5E7EB` 边框，6px 圆角，13px 等宽字体
- 表格：表头 `#F3F4F6` 背景，边框 `#E5E7EB`，行高 40px
- 列表：1.5em 缩进，8px 行间距
- 链接：主色 `#4F46E5`，hover 下划线

#### C3. 应用场景

| 场景 | 实现方式 | 优先级 |
|------|----------|--------|
| 项目摘要弹窗 | 替换现有 Modal 纯文本为 MarkdownPreview | P0 |
| .md 文件预览 | 文件列表新增「预览」按钮，点击弹出 Modal | P1 |
| 转交包 context.md | 导入预览中渲染转交说明 | P1 |
| AI 对话渲染 | 复用同一组件，优化样式 | P2 |

#### C4. 文件预览入口

在 `FileListTable` 的文件操作列中，对 `.md` 类型文件新增「预览」按钮：

```
📄 README.md    [预览] [打开] [分类] [删除]
```

点击后弹出全屏 Modal，使用 MarkdownPreview 组件渲染。

**新增依赖：** `rehype-highlight`、`rehype-raw`。

## 五、风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| AI 文件推荐不准确 | 协作转交体验差 | 提供手动调整入口，AI 只辅助不决定；推荐理由帮助用户判断 |
| 转交包体积过大 | 微信/钉钉传文件有大小限制 | 协作转交本身是精简包；整体转交时在对话框显示预估大小，超过 100MB 提示 |
| Markdown 渲染 XSS | `rehype-raw` 可能执行恶意 HTML | 配合 `rehype-sanitize` 过滤危险标签 |
| 跨版本导入兼容 | 旧版无法导入新版转交包 | `format_version` 字段校验 + 向前兼容的 JSON 结构（新字段 optional） |
| 同名项目冲突 | 导入时覆盖已有项目 | 导入对话框提示冲突，默认自动重命名，不自动覆盖 |

## 七、推迟至后续版本的功能

以下功能从 v0.2.0 草稿中移除，在后续版本中按需排入：

| 功能 | 推迟原因 | 建议版本 |
