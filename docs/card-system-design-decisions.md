# PMAer 卡片系统设计决策记录

> 版本：0.2.0 | 创建时间：2026-06-23
> 用途：记录卡片系统每个设计决策的业务原因和考量，方便未来优化时有据可依

---

## 一、通用统计行

### 1.1 里程碑替代关键问题

**决策**：统计行第二张卡片从"关键问题"改为"里程碑"

**业务原因**：关键问题已在进行中特色卡片中有独立展示，统计行不需要重复。里程碑是项目经理日常最关注的信息——"下一步要干什么"——放在统计行让每次进入项目详情页都能一眼看到。

**展示内容**：下一步里程碑名称 + 日期，点击弹窗查看完整时间轴。

---

## 二、里程碑卡片

### 2.1 两类里程碑合并展示

**决策**：项目里程碑和付款里程碑合并到同一个时间轴中展示，同一个项目阶段的多个合同（实施/开发）的付款信息并列标注。

**业务原因**：一个项目通常包含实施合同和开发合同，两者的付款节点往往对应同一个项目里程碑（如"上线确认"）。合并展示避免用户在多个地方找信息，同一个节点上能直接看到实施和开发分别的付款金额。

**示例**：上线确认节点显示"实施 ¥7.26万(20%) / 开发 ¥1.50万(20%)"。

### 2.2 合同签订作为里程碑

**决策**：合同签订（首付款节点）也作为里程碑记录。

**业务原因**：PMAer 是项目全生命周期管理工具，合同签订是项目正式启动的关键节点。虽然没有"交付物"，但它是一个明确的付款触发点（通常50%首付款），需要在时间轴上体现。

### 2.3 手动完成与编辑锁定

**决策**：里程碑支持手动勾选完成和编辑日期，手动编辑后的里程碑不受后续 AI 识别覆盖。

**业务原因**：文件上传识别里程碑完成状态存在两个问题——① 文件上传可能滞后（实际完成了但文件没及时传），② AI 识别可能不准确。需要给项目经理手动操作的入口。同时，手动修改后的数据应保持稳定，避免 AI 重新分析时把人工确认的信息覆盖掉。

**实现**：`manuallyEdited` 标志位为 true 时，AI 合并逻辑跳过 `date`、`completed`、`completedDate` 字段。

### 2.4 付款状态使用"已确认"而非"已付款"

**决策**：付款里程碑的状态用"已确认"/"待确认"，不用"已付款"/"待付款"。

**业务原因**：项目经理能获取的信息是签字确认单是否签署，这代表"甲方确认了这个阶段的交付物，认可付款条件"。但实际打款是财务层面的操作，项目经理通常不掌握。用"已确认"准确反映了项目管理层面能追踪到的粒度。

### 2.5 里程碑按项目阶段分组

**决策**：所有里程碑标注所属阶段（启动/方案/构建/测试/上线/验收），弹窗中按阶段分组展示。

**业务原因**：方便快速定位项目进展到哪个阶段。如"业务蓝图确认"和"开发设计确认"都属于方案阶段的关键节点，放在一起便于对比。

**决策**：所有里程碑标注所属阶段（启动/方案/构建/测试/上线/验收），弹窗中按阶段分组展示。

**决策**：里程碑支持手动勾选完成和编辑日期，`manuallyEdited=true` 时 AI 不覆盖 `date`/`completed`/`completedDate`。

**业务原因**：文件上传可能滞后，AI 识别可能不准确，需要手动入口。手动确认后数据应保持稳定。

## 三、合同概览卡片

### 3.1 从签字文件列表重新定位为合同全貌

**决策**：合同概览从"合同文件列表+签字状态"重新设计为"合同金额+分项明细+付款里程碑+已确认收入"。

**业务原因**：原设计（合同文件列表）与签字追踪卡片功能重复。合同概览的核心价值是让项目经理快速掌握"这个项目合同多少钱、各部分多少、确认了多少收入"。签字文件追踪应独立处理项目关键节点的交付物签字状态。

### 3.2 展示合同分项（软件/实施/开发/...）

**决策**：按合同组成部分分行展示名称、描述、金额、人天。

**业务原因**：一个项目合同通常包含多个组成部分（软件许可、实施服务、客户化开发、生态软件等），各部分金额和人天不同。项目经理在对外沟通、内部汇报、续签评估时经常需要这些分项数据。从 SOW 和合同文件中可以提取到这些信息。

### 3.3 已确认收入与付款里程碑关联

**决策**：已确认收入 = 所有 `confirmed = true` 的付款里程碑的金额之和。

**业务原因**：项目经理需要知道"合同总额中已经确认了多少"，这是项目商务进度的核心指标。确认收入基于里程碑签字确认，而非实际到账（实际到账是财务数据，项目经理通常不掌握）。

### 3.4 合同概览关联付款里程碑

**决策**：合同概览的弹窗中展示付款里程碑时间线，每个节点显示实施和开发分别的金额和确认状态。

**业务原因**：付款里程碑本质上是合同约定的付款条件，和合同概览是"整体-明细"的关系。在合同概览中能直接看到付款进度，不需要跳转到其他地方。

## 四、项目信息卡片

### 4.1 灵活的 key-value 结构

**决策**：项目信息采用可扩展的 key-value 结构，核心字段 AI 提取，扩展字段可持续增加。

**业务原因**：不同项目需要填写的关键信息不完全相同（验收确认单、开工报告、结项报告等文档格式有差异）。flexible 的结构允许逐步增加字段，不需要每次都改代码。AI 从合同/SOW 中提取关键信息，减少项目经理手填量。

### 4.2 手填字段明确标注

**决策**：必须手填的字段（如我方项目经理、客户项目经理）始终展示"填写"入口。

**业务原因**：这些信息无法从文件中提取（通常是口头约定或组织内部指定），但后续文档填写时必须要用。

## 五、特色卡片整体规则

### 5.1 侧边栏驱动卡片切换

**决策**：特色卡片根据侧边栏选中阶段自动切换，不需要单独的切换按钮。

**业务原因**：侧边栏本身就是阶段导航，卡片跟随切换是自然的交互逻辑。用户点击"售前"看售前相关卡片，点击"需求"看进行中相关卡片，无需额外操作。

### 5.2 阶段分组与特色卡片映射（更新于 2026-06-25）

**决策**：特色卡片分为 4 组，按阶段映射：

| 分组 | 项目阶段 | 特色卡片 |
|------|----------|----------|
| presale（售前） | 售前 | 项目信息 + 合同概览 + 项目评估 |
| progress（进行中） | 启动、需求、方案、构建、测试、上线、验收 | 需求跟踪 + 关键问题 + 签字追踪 |
| handover（转客户成功） | 转客户成功 | 签字追踪 + 拓展商机 + 交付物清单 |
| closed（关闭） | 关闭 | 拓展商机 + 交付物清单 + 项目总结 |

**业务原因**：转客户成功阶段的核心关注点是交接与收尾（确认签字完成、记录后续商机、整理交付物），与进行中的"执行管理"和关闭的"归档复盘"都不同，需要独立的卡片组合。

### 5.3 启动归入进行中

**决策**：启动阶段归入进行中分组，不单独一组。

**业务原因**：启动和需求/方案/构建/测试/上线/验收都属于项目执行阶段，卡片内容一致（需求跟踪+关键问题+签字追踪）。单独分组没有信息增量。

### 5.4 里程碑只在统计行，不重复出现在特色卡片

**决策**：里程碑信息只在统计行展示（概览+弹窗），特色卡片区域不重复显示。

**业务原因**：避免信息冗余。统计行的里程碑已经足够用户获取项目进度信息，特色卡片区域应聚焦于每个阶段特有的业务信息。

## 五、项目评估卡片

### 5.1 内置利润测算工具

**决策**：项目评估卡片内置完整的利润测算功能，可添加多个内部成员（不同角色+职级）。不做分包/类整包模式切换，统一适用同一套红线规则。

**业务原因**：Excel 利润测算表公式复杂，项目经理手动计算容易出错。简化版去掉分包/类整包模式切换，因为两者利润率红线一致（内部>0%、外包>40%），区别仅在于自身交付成本占比参考红线不同（5% vs 10%），这个差异通过展示标签体现即可。

### 5.2 红线标准（统一）

**决策**：所有模式统一适用以下红线：

| 指标 | 红线 | 说明 |
|------|------|------|
| 内部利润率 | < 0% | 自有交付不能亏本 |
| 外包利润率-自有产品 | < 40% | 触发低利润审批 |
| 整包：自身交付成本占比 | ≤ 5% | 参考线，仅展示 |
| 类整包：自身交付成本占比 | ≤ 10% | 参考线，仅展示 |

**业务原因**：利润率红线在所有外包形式下一致。成本占比的参考线（5%/10%）作为展示信息提供，不影响利润率计算。

### 5.4 内部人天单价表

**决策**：从成本表提取完整的角色×职级单价表（实施顾问/开发工程师/项目经理/测试工程师 C1-1 到 C4-1，运维工程师服务1-1/1-2），内置到系统中。

**业务原因**：单价表是公司内部数据，项目经理不一定都记得。内置后减少查表时间，也避免填错。

### 5.5 算法文档化

**决策**：利润测算的完整算法记录在 `docs/evaluation-algorithm.md`，供 AI 对话时调用。

**业务原因**：未来 AI 对话中用户说"帮我算一下这个项目的利润率"时，AI 可以直接引用算法文档计算结果并判断红线，无需用户逐个数字手动测试。

### 5.6 两个版本

**决策**：卡片版（PMAer 内嵌，480px 紧凑布局）和网页版（独立分享，1040px 宽屏双栏布局）分别实现。网页版作为独立工具通过微信/邮件分享。

**业务原因**：卡片版需要适配 PMAer 的卡片网格，空间有限。网页版不需要安装 PMAer，任何浏览器打开即可使用，方便分享给不使用 PMAer 的同事。

### 5.7 差旅费归属

**决策**：差旅费拆为"内部差旅"和"外部差旅"两个独立字段，分别计入内部总成本和外包总成本，直接影响利润率计算。

**业务原因**：Excel 原始模板中差旅费是每个资源行独立的字段，内部和外包分别有差旅费。简化版拆为两个字段，既保持了计算准确性（差旅费影响利润率），又避免了让每个成员都填差旅费的复杂度。内部差旅加到内部人天成本上得到内部总成本，外包差旅加到外包人天成本上得到外包总成本。

## 六、需求跟踪卡片

### 6.1 定位

**决策**：需求跟踪卡片定位为"需求全生命周期跟踪工具"，不只是记录需求，而是跟踪需求从提出到关闭的完整过程。

**业务原因**：项目经理日常需要跟踪多个需求的进展，需求可能跨多个文件（需求文档、会议纪要、周报）被多次提及和更新。需要一个地方聚合所有需求信息，看到每个需求的全周期记录。

### 6.2 数据模型

**决策**：采用 Requirement + TimelineEntry 两层结构，存储在 `project.metadata.requirements` 数组中。

```
Requirement {
  id: string
  name: string                    // 需求名称
  status: 'pending'|'progress'|'completed'|'delayed'|'closed'
  assignee?: string               // 跟进人
  expectedEnd?: string            // 预期结束时间
  solution?: string               // 需求方案
  result?: string                 // 解决结果
  source?: string                 // 来源文件名
  createdAt: string
  updatedAt: string
  timeline: TimelineEntry[]       // 全生命周期时间轴
}

TimelineEntry {
  id: string
  date: string
  title: string                   // 事件标题
  content: string
  source?: string
}
```

**业务原因**：需求不是一次性记录，而是随着项目推进不断更新。时间轴结构让每个需求有完整的生命周期记录，从需求确认到方案讨论到进度更新到问题记录。

### 6.3 UI 三层结构

**决策**：卡片概览（3 条最新更新） → 弹窗列表（全部需求） → 行内展开详情（完整信息 + 时间轴）。

**业务原因**：三层结构既保证了概览的简洁，又提供了完整的详情查看能力。行内展开（而非第三层弹窗）避免了多层弹窗的复杂度。

### 6.4 数据来源与 AI 匹配

**决策**：需求数据有两个入口——AI 自动提取 + 手动录入。AI 提取时需要做名称匹配：匹配上的追加 timeline，匹配不上的创建新需求。

**业务原因**：同一需求会跨多个文件被多次提及（需求文档确认一次、会议纪要讨论一次、周报更新一次），如果不做匹配会产生大量重复条目。AI 按名称模糊匹配后追加 timeline，保证每个需求有完整的生命周期记录。

### 6.5 状态管理

**决策**：5 种状态——pending（待确认）、progress（进行中）、completed（已完成）、delayed（延期）、closed（已关闭）。关闭操作在需求详情中执行，关闭后卡片概览默认不显示。

**业务原因**：需求从提出到关闭有明确的状态流转。closed 状态用于归档已完成且不再跟踪的需求，避免概览信息过载。

### 6.6 性能评估

**决策**：需求数据存储在 `project.metadata.requirements` JSON 字段中。50 条需求每条 5 条时间轴约 125KB，远低于性能阈值。弹窗列表不需要虚拟滚动。

**业务原因**：需求数据量级（几十条）和时间轴记录量级（每条几到十几条）不会对 SQLite 或 DOM 渲染产生性能影响。

## 七、关键问题卡片

### 7.1 定位

**决策**：关键问题卡片定位为"项目风险点跟踪工具"，与需求跟踪卡片结构相似但侧重不同——需求关注"要做的事"，关键问题关注"阻碍项目的风险点"。

**业务原因**：项目经理日常需要跟踪多个风险问题，问题有明确的提出人和解决人，需要在解决后反馈提出人。问题的生命周期比需求更短但更紧急。

### 7.2 数据模型

```
Issue {
  id: string
  title: string                   // 问题标题
  description?: string            // 问题描述
  priority: 'high'|'medium'|'low' // 优先级（支持手动调整）
  status: 'open'|'analyzing'|'resolved'|'closed'
  reporter?: string               // 提出人（谁发现的）
  resolver?: string               // 解决人（谁来解）
  source?: string                 // 来源文件名
  createdAt: string
  resolvedAt?: string             // 解决时间
  confirmedByReporter?: boolean   // 是否已反馈提出人
  timeline: TimelineEntry[]       // 跟踪记录
}
```

**业务原因**：问题需要明确提出人（谁发现的）和解决人（谁来解），不同于需求的跟进人概念。优先级支持手动调整以反映实际紧急程度变化。

### 7.3 UI 结构

**决策**：与需求跟踪相同的三层结构——卡片概览（3 条高优先级问题）→ 弹窗列表 → 行内展开详情+时间轴。

**业务原因**：卡片概览优先展示高优先级（high）问题，让项目经理一眼看到最紧急的风险。

### 7.4 状态管理

**决策**：4 种状态——open（待处理）、analyzing（分析中）、resolved（已解决）、closed（已关闭）。关闭操作在详情中执行。

**业务原因**：问题不需要 closed 后的"反馈提出人"自动通知（当前版本不做），但保留 `confirmedByReporter` 字段供未来扩展。手动标记已反馈即可。

### 7.5 优先级管理

**决策**：优先级（high/medium/low）支持手动调整，通过下拉菜单切换。

**业务原因**：问题的紧急程度会随项目推进变化（如原来 high 的问题有了替代方案降为 medium），手动调整让数据始终反映真实状态。

## 八、签字追踪卡片

### 8.1 定位

**决策**：签字追踪卡片定位为"项目关键确认文件跟踪工具"，基于项目金额和内容自动生成签字文件清单，支持手动增减和文件上传。

**业务原因**：项目在不同阶段需要客户/内部签字确认的文件不同，且数量取决于项目金额档位和是否包含开发内容。需要一个工具自动判断需要哪些确认单，减少遗漏。

### 8.2 签字文件生成规则

**决策**：根据项目金额确定档位，再根据是否有开发内容过滤，自动生成签字文件清单。

| 档位 | 金额范围 | 必选文件数 | 说明 |
|------|---------|-----------|------|
| A | 50万以上 | 19 | 最高要求 |
| B | 50万以下 | 8 | 标准要求 |
| C | 10万以下 | 5 | 最低要求 |
| 默认 | 金额不确定 | 5（按C档） | 取最低要求 |

**过滤规则**：如果项目合同分项中不包含开发服务，则去掉"开发需求设计确认单"和"开发完成确认单"。

**业务原因**：不同金额和内容的项目，需要签字确认的文件不同。自动判断减少了项目经理的遗漏风险。金额不确定时按最低要求，宁可少要求不遗漏。

### 8.3 数据模型

```
SignatureDoc {
  id: string
  name: string                    // 文件名称
  required: boolean              // 是否必须（根据档位判断）
  status: 'unsigned'|'signed'    // 签字状态
  signer?: string                // 签字人
  signedAt?: string              // 签字日期
  fileId?: string                // 关联的已上传文件ID
  source: 'auto'|'manual'       // 自动生成 or 手动添加
}
```

**业务原因**：区分 `source` 字段是为了知道哪些是系统推荐的必须文件，哪些是用户额外添加的。`fileId` 关联到 files 表，上传签字文件后自动关联。

### 8.4 UI 结构

**决策**：卡片概览（进度条 + 前3个文件）→ 弹窗列表（全部文件 + 上传/查看按钮）。

**业务原因**：签字文件数量通常 5-19 个，卡片概览用进度条+文件列表展示，弹窗中支持手动增减和上传操作。每个文件行后有「上传」按钮，用户可以直接上传签字文件，无需先通过 AI 判断。

### 8.5 文件上传联动

**决策**：签字文件列表中的每个文件行有「上传」按钮，点击后上传文件并自动关联到对应行，更新签字状态。如果 AI 识别到签字（has_signature），自动将状态改为 signed。

**业务原因**：项目经理拿到签字确认单后可以直接在卡片中上传，省去了先上传到文件列表再手动关联的步骤。AI 签字检测能力已在现有系统中实现，可复用。

## 九、拓展商机卡片

### 9.1 定位

**决策**：拓展商机卡片定位为"未来项目拓展备忘工具"，记录项目中提到的二期三期规划、客户追加需求等，供未来项目参考。不做时间轴跟踪。

**业务原因**：项目执行过程中经常有客户追加需求、未来优化约定等，这些信息在项目结束后容易遗忘。需要一个轻量级的备忘录记录客户提了什么、讨论了什么方案，但不需要像需求跟踪那样做全生命周期管理。

### 9.2 数据模型

```
Opportunity {
  id: string
  title: string                   // 商机标题
  description?: string            // 客户需求描述
  solution?: string               // 讨论的方案
  phase?: string                  // 规划阶段（二期/三期/远期）
  status: 'planned'|'confirmed'|'dropped'  // 规划中/已确认/已放弃
  source?: string                 // 来源文件名
  createdAt: string
  updatedAt: string
}
```

**业务原因**：与需求/问题卡片相比，拓展商机更轻量——没有时间轴、没有跟进人/解决人、状态只有 3 种。核心字段是 description（客户提了什么）和 solution（我们打算怎么做），为未来项目提供参考。

### 9.3 UI 结构

**决策**：卡片概览（3 条）→ 弹窗列表。无行内展开详情。

**业务原因**：商机信息量不大（标题+需求描述+方案），弹窗列表中直接展示完整信息，不需要额外的展开层级。

## 十、交付物清单卡片

### 10.1 定位

**决策**：交付物清单卡片定位为"项目交付成果版本管理工具"，跟踪每个交付物（蓝图/手册/脚本/方案等）的版本演进，不关联签字件。

**业务原因**：交付物和签字件是独立的两件事——交付物是实际工作成果，签字件是确认单。有些项目不需要签字确认但仍需要交付成果。交付物的核心价值在于版本管理（草稿→整合→待交付→已交付），签字状态由签字追踪卡片独立管理。

### 10.2 数据模型

```
Deliverable {
  id: string
  name: string                    // 交付物名称
  type: string                    // 类型（蓝图/手册/脚本/方案/...）
  status: 'draft'|'merged'|'ready'|'delivered'
  currentVersion: string          // 当前版本号
  versions: Version[]             // 版本历史
  createdAt: string
  updatedAt: string
}

Version {
  id: string
  versionNo: string               // 版本号（v1.0, v2.0, 最终版）
  status: 'draft'|'merged'|'final'
  fileId?: string                 // 关联的文件ID
  createdAt: string
  note?: string                   // 版本说明
}
```

**业务原因**：版本管理是交付物卡片的核心。每个交付物可能有多个版本（草稿→整合版→最终版），需要记录版本历史和关联的文件。不关联签字件以保持卡片职责单一。

### 10.3 UI 结构

**决策**：卡片概览（3 条，显示名称+状态+当前版本）→ 弹窗列表（全部交付物 + 版本历史展开）。

**业务原因**：交付物数量通常 5-10 个，弹窗中展示完整版本历史。每个版本有「查看」按钮可打开关联文件。

## 十一、项目总结卡片

### 11.1 定位

**决策**：项目总结卡片分为两个部分——"项目成果"（面向外部汇报/分享）和"项目复盘"（面向内部经验沉淀）。

**业务原因**：项目总结不是给自己看的，是给未来汇报、分享用的。成果部分用于上线汇报、内部分享；复盘部分用于经验沉淀、改进参考。

### 11.2 数据模型

```
ProjectSummary {
  achievements: Achievement[]     // 项目成果
  retrospective: Retrospective    // 项目复盘
  updatedAt: string
}

Achievement {
  id: string
  title: string                   // 成果标题
  category: 'feature'|'recognition'|'team'|'metric'|'highlight'
  description: string             // 描述
  data?: string                   // 量化数据
  source?: string                 // 来源文件
}

Retrospective {
  highlights: string[]            // 做得好的
  improvements: string[]          // 待改进的
  lessons: string[]               // 经验教训
  qualityIssues: QualityIssue[]   // 开发质量问题
}
```

### 11.3 UI 结构

**决策**：卡片概览（3 条成果摘要 + 统计）→ 弹窗详情（两个 Tab：项目成果 / 项目复盘）。

**业务原因**：成果和复盘面向不同受众，用 Tab 分开。提供"复制为汇报格式"和"复制为PPT大纲"两种复制格式。

## 十二、AI 摘要卡片

### 12.1 定位

**决策**：AI 摘要放在统计行第四张卡片，点击弹窗查看完整 Markdown 渲染内容。

**业务原因**：AI 摘要是项目全局信息的浓缩，项目经理进入详情页时可能想快速了解项目全貌。放在统计行让每次进入都能看到"查看摘要"入口。

### 12.2 数据来源

**决策**：数据源为 `.ai/project-summary.md` 文件，由 AI 分析所有项目文件后生成。

**业务原因**：摘要由 AI 基于项目全部文件上下文生成，包含项目概况、文件列表、进度、关键问题、建议等。

### 12.3 UI 结构

**决策**：统计行卡片显示"查看摘要"+"生成/更新"两个按钮。点击"查看摘要"打开 800px 宽弹窗，使用 MarkdownPreview 组件渲染。

**业务原因**：摘要内容通常较长（Markdown 格式），需要较大弹窗展示。复用模块 C 的统一 Markdown 渲染组件。
