**总体评价：粒度基本合适，但有5处需要调整**

### 1.2 需要调整的5处

"项目计划"（项目计划书、里程碑、WBS、甘特图）从"需求"阶段移出。

> **【用户决策】** 项目计划在启动时就要有，后续文件会持续更新，因此更适合放在「启动」阶段而非"方案"阶段。

#### ② 上线阶段：合并"上线切换方案"与"部署文档" ✅ 已接受

上线阶段有7个子分类，是全表最多的，其中存在语义重叠：

- "上线切换方案"和"部署文档"：都是部署执行相关的文档，区别仅在于"方案"vs"执行"，AI很难准确区分。合并为"部署方案与文档"。

> **【用户决策】** 接受合并。

合并后上线阶段变为6个子分类：部署方案与文档、操作手册、签字材料、初始化材料、问题清单、会议纪要。

#### ③ 验收阶段："项目总结"移至「关闭」阶段 ✅ 已接受

验收阶段的"项目总结"与关闭阶段的"复盘总结"语义重叠。

> **【用户决策】** 接受调整。将"项目总结"移至"关闭"阶段，与"复盘总结"合并为"项目总结与复盘"。验收阶段只保留3个子分类：验收材料（待签）、验收签字件（已签）、验收报告。关闭阶段变为2个子分类：项目归档、**项目总结与复盘**。

#### ④ 构建阶段：不新增"架构与设计"子分类 ✅ 无需调整

原审核建议增加"架构与设计"子分类。

> **【用户决策】** 不需要单独建分类。蓝图子分类覆盖实施侧的设计（业务蓝图、流程设计），开发规格说明书覆盖开发侧的架构与设计（技术选型、架构设计、接口规格）。两者已涵盖架构与设计类文档，AI分类时应能正确归入这两个子分类。

#### ⑤ 售前阶段：合并"成本评估"与"报价单" → "商务报价"

原审核指出"报价单"和"成本评估"边界模糊，AI分类时容易混淆。

> **【用户决策】** 接受合并。将"报价单"和"成本评估"合并为"商务报价"子分类，涵盖报价清单、费用明细、人天估算、成本估算、利润分析、资源评估等所有商务财务类文档。

## 二、验收阶段"待签/已签"状态区分

**审核结论：业务需求合理，但当前技术实现存在3个问题**

### 2.1 当前签字检测的技术限制

当前 `files` 表使用 `has_signature BOOLEAN DEFAULT FALSE` 存储签字检测结果。这个字段存在语义歧义：`false` 同时代表"未检测"和"检测后无签字"两种状态。

文档要求的"待签/已签"是一个**流程状态**（pending → signed），而签字检测是**内容检测**（有签字 / 无签字）。这是两个不同的概念，不能简单等同。

### 2.2 建议的技术方案

新增独立的 `signature_status` 字段（TEXT类型），替代现有的 `has_signature` 布尔字段：

```sql
-- 迁移：新增 signature_status 字段
ALTER TABLE files ADD COLUMN signature_status TEXT DEFAULT NULL;

-- 数据迁移：将现有 has_signature 映射到新字段
UPDATE files SET signature_status = CASE
  WHEN has_signature = 1 THEN 'signed'
  WHEN has_signature = 0 AND content_extracted IS NOT NULL THEN 'unsigned'
  ELSE NULL
END;
```

字段取值：
- `NULL`：不需要签字状态（非验收材料类文件）
- `'pending'`：验收材料，需要签字但尚未检测到签字
- `'signed'`：检测到签字
- `'unsigned'`：已检测确认无签字

### 2.3 实现要点

1. **AI分类时判断**：当文件被分类到"验收材料（待签）"子分类时，自动设置 `signature_status = 'pending'`
2. **签字检测后更新**：`SignatureDetector` 检测完成后，更新 `signature_status` 为 `'signed'` 或 `'unsigned'`
3. **前端展示**：文件列表中用不同颜色Tag显示签字状态（灰色=待签，绿色=已签，红色=无签字）
4. **手动修改**：用户可手动切换签字状态（应对AI检测不准的场景）
5. **保留 `has_signature` 字段**：向后兼容，不删除旧字段，`signature_status` 作为新的权威字段

### 2.4 需要额外确认的问题

- 签字检测当前仅支持PDF和图片。对于 `.doc` / `.docx` 格式的验收材料，`signature_status` 如何处理？建议：非PDF/图片的验收材料默认为 `'pending'`，提示用户手动确认。

## 三、会议纪要跨阶段方案

**审核结论：方案方向正确，但Prompt设计需要加强**

### 3.1 当前方案的评估

文档提出"会议纪要保留在各阶段内，AI通过会议内容判断阶段归属"。这个方案比单独建一个"会议纪要"类别更好，因为：
- 同一份会议纪要往往只与一个阶段相关
- 按阶段分组更符合用户的查找习惯
- 避免了"会议纪要"文件夹下混杂不同阶段的会议

### 3.2 需要在Prompt中明确的3件事

**第一，会议纪要的特征识别规则**

当前 `classify.ts` 的Prompt模板没有明确告诉AI如何识别会议纪要。建议增加：

```
会议纪要识别特征（满足任一即为会议纪要）：
1. 文件名包含"纪要"、"会议记录"、"meeting minutes"、"MOM"
2. 文件内容前200字包含"会议时间"、"参会人员"、"会议议题"等会议格式关键词
3. 文件内容以"会议主题"或"一、会议背景"开头
```

**第二，会议纪要的阶段归属判断逻辑**

当识别到会议纪要后，如何判断它属于哪个阶段？建议在Prompt中增加：

```
会议纪要阶段归属规则（按优先级排序）：
1. 文件名或内容中明确提到阶段名称 → 归入该阶段
   例："需求评审会议纪要" → 需求/会议纪要
   例："上线协调会" → 上线/会议纪要
2. 如果无法从内容判断 → 参考同批次上传的其他文件的阶段
3. 如果仍然无法判断 → 归入"启动/启动会议"（兜底）
```

**第三，跨阶段会议纪要的处理**

实际业务中，一次会议可能同时讨论需求和方案。建议在Prompt中明确：
- AI只判断**主要阶段**（以讨论内容最多的阶段为准）
- 如果AI无法确定主阶段，输出 `confidence < 0.5`，由用户手动选择

### 3.3 UI展示建议

左侧边栏的"会议纪要"子分类可以考虑加一个小标记，显示该阶段下有多少份会议纪要。例如：`需求 > 会议纪要 (3)`。

## 四、数据库schema变更 + 文件夹二级结构的技术可行性

**审核结论：技术上可行，但改动范围大（至少11个文件），需要分步实施**

### 4.1 数据库schema变更

**改动量：中等**

需要修改的文件：

| 文件 | 改动内容 |
|------|---------|
| `electron/database/index.ts` | files表schema添加 `sub_category TEXT` 字段 |
| `electron/database/index.ts` | 添加迁移脚本（ALTER TABLE） |
| `electron/database/files.ts` | FileRecord接口添加 `sub_category`；ALLOWED_FILE_FIELDS白名单更新；INSERT语句更新 |
| `src/types/index.ts` | 前端FileRecord接口添加 `sub_category` |
| `src/types/windowApi.ts` | IPC接口类型更新 |

**索引建议**：当前有 `idx_files_category(project_id, category)` 索引，建议扩展为 `(project_id, category, sub_category)` 复合索引，用于按阶段+子分类的联合查询。

### 4.2 文件夹二级结构

**改动量：大（核心改动）**

这是最复杂的部分。当前文件路径为 `项目目录/阶段名/文件名`，需要改为 `项目目录/阶段名/子分类名/文件名`。

需要修改的位置：

| 文件 | 行号 | 改动内容 |
|------|------|---------|
| `electron/ipc/file-handlers.ts` | 19 | `VALID_CATEGORIES` 从字符串数组改为 `{ 阶段: string[] }` 的映射结构 |
| `electron/ipc/file-handlers.ts` | 29-49 | `moveFileToCategory()` 函数添加 `subCategory` 参数 |
| `electron/ipc/file-handlers.ts` | 174-183 | AI分类后的文件移动逻辑：`path.join(projectPath, category, subCategory)` |
| `electron/ipc/file-handlers.ts` | 303-368 | `file:updateCategory` 处理二级路径 |
| `electron/prompts/classify.ts` | 全文 | Prompt中添加子分类描述和返回格式 |
| `electron/utils/ai-response.ts` | 16-17 | `parseClassifyResponse` 添加 `subCategory` 解析 |

### 4.3 向后兼容：最大风险点

**已有项目使用一级文件夹结构，新项目使用二级结构，会导致两套体系共存。** 需要明确迁移策略：

**建议方案：渐进式迁移**

1. 新建项目：使用二级结构（阶段/子分类）
2. 已有项目：保持一级结构不变
3. 已有项目重新分类时：如果用户手动触发"一键分类"，将文件从一级结构迁移到二级结构
4. 项目设置中增加"启用子分类"开关，默认为开启（新项目）

**迁移脚本需要处理的问题：**
- 遍历一级文件夹下的所有文件
- 根据数据库 `category` + `sub_category` 字段计算新路径
- 创建二级子目录
- 移动文件
- 更新 `stored_path` 字段
- 处理文件名冲突

### 4.4 AI分类Prompt的复杂度

当前Prompt只要求AI从11个分类中选一个。加入子分类后，AI需要从约40个选项中选择。这会增加Prompt长度和AI判断难度。

**建议优化：**
- 采用两阶段分类：第一阶段判断大分类（11选1），第二阶段判断子分类（3-7选1）
- 或者：一次请求，让AI同时返回 `category` 和 `sub_category`，但在Prompt中用结构化格式清晰列出每个阶段的子分类选项

推荐第二种方案（一次请求），因为两次请求会翻倍AI调用成本。

### 4.5 实施路径建议

建议分3步实施，每步独立可验证：

**Step 1：数据库 + 类型定义（1-2小时）**
- 添加 `sub_category` 字段到 files 表
- 更新前后端类型定义
- 添加复合索引

**Step 2：AI分类 + Prompt更新（2-3小时）**
- 更新 classify Prompt，添加子分类描述
- 更新 `parseClassifyResponse` 解析逻辑
- 更新 `moveFileToCategory` 支持二级路径

**Step 3：前端UI + 文件夹迁移（3-4小时）**
- 文件列表表格显示子分类列
- 手动分类下拉框支持子分类选择
- 新项目创建二级文件夹结构
- 已有项目的迁移逻辑
