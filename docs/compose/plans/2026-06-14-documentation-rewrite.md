## 一、核心概念说明（重要）

### 1.1 两套阶段体系

本系统存在**两套独立的阶段体系**，用途不同，不可混淆：

#### 项目阶段（Project Stage）— 3个

用于**项目整体状态管理**，表示项目当前所处的宏观阶段：

| 阶段 | 含义 | 触发条件 |
|------|------|----------|
| **售前** | 项目尚未正式立项，处于商机跟踪阶段 | 创建项目时默认 |
| **进行中** | 项目已立项，正在执行中 | AI识别到合同/启动文件时自动推进，或用户手动推进 |
| **关闭** | 项目已完成交付，归档状态 | AI识别到验收/上线文件时自动推进，或用户手动推进 |

**设计原则：**
- 项目阶段保持**粗粒度**（仅3个），因为：
  1. 过细的阶段划分会增加AI判断复杂度
  2. 项目管理本身不需要精确到每个执行步骤
  3. 精细化分类由「文件分类阶段」承担

#### 文件分类阶段（File Classification Stage）— 11个

用于**文件精细化分类**，表示文件内容属于项目生命周期的哪个具体环节：

| 阶段 | 含义 | 典型文件 |
|------|------|----------|
| **售前** | 商机阶段文件 | 投标书、方案报价、客户需求调研 |
| **启动** | 项目立项后启动阶段 | 立项通知、启动会议纪要、项目章程 |
| **需求** | 需求调研与分析阶段 | 需求规格说明书、调研报告、用户故事 |
| **方案** | 技术方案设计阶段 | 技术方案、架构设计、UI设计稿 |
| **构建** | 开发实施阶段 | 代码文档、接口文档、开发规范 |
| **测试** | 测试验证阶段 | 测试报告、缺陷清单、测试用例 |
| **上线** | 部署上线阶段 | 部署手册、上线清单、运维文档 |
| **验收** | 客户验收阶段 | 验收报告、签字文件、验收清单 |
| **转客户成功** | 交付后交接阶段 | 培训资料、交接清单、运维手册 |
| **关闭** | 项目归档阶段 | 项目总结、归档清单 |
| **未分类** | 尚未被AI分析的文件 | 新上传的文件默认状态 |

**设计原则：**
- 文件分类阶段保持**细粒度**（11个），因为：
  1. 文件内容需要精确分类以便检索
  2. 不同阶段的文件有不同处理逻辑
  3. AI需要更精细的判断来准确分类

### 1.2 两套阶段的关系

```
项目阶段（3个）          文件分类阶段（11个）
┌─────────────┐         ┌─────────────────────┐
│   售前      │ ←─────→ │ 售前、启动           │
├─────────────┤         ├─────────────────────┤
│   进行中    │ ←─────→ │ 需求、方案、构建、    │
│             │         │ 测试、上线、验收、    │
│             │         │ 转客户成功           │
├─────────────┤         ├─────────────────────┤
│   关闭      │ ←─────→ │ 关闭                 │
└─────────────┘         └─────────────────────┘
```

**关键区别：**
- 项目阶段是**项目级**属性，一个项目只有一个当前阶段
- 文件分类阶段是**文件级**属性，每个文件有一个分类阶段
- AI分类文件时，会同时判断文件的分类阶段，并可能触发项目阶段推进

### 1.3 阶段推进规则

当AI分类文件时，如果识别到文件属于更高阶段，会弹窗询问是否推进项目阶段：

| 当前项目阶段 | 文件分类阶段 | 是否触发推进询问 |
|--------------|--------------|------------------|
| 售前 | 售前、启动 | 否 |
| 售前 | 需求、方案、构建、测试、上线、验收、转客户成功、关闭 | **是** → 询问是否推进到「进行中」 |
| 进行中 | 需求、方案、构建、测试、上线、验收、转客户成功 | 否 |
| 进行中 | 关闭 | **是** → 询问是否推进到「关闭」 |
| 关闭 | 任何阶段 | 否（已归档） |

## 二、产品定位与目标用户

### 2.1 产品定位

**项目管理助手**是一款面向项目管理人员的 Windows 桌面端 AI 驱动工具。核心价值：

1. **集中管理**：将散落在邮件、U盘、微信中的项目文件集中到统一位置
2. **智能分类**：AI 自动识别文件内容，归类到对应的文件分类阶段（11个）
3. **阶段感知**：通过文件内容自动判断项目整体进展（3个项目阶段）
4. **AI 助手**：基于项目全部文件上下文，提供分析、问答、摘要能力

### 2.2 目标用户与角色

| 角色 | 典型场景 | 关注重点 |
|------|---------|---------|
| **项目经理（PM）** | 管理多个客户项目的文件、跟进进度、协调团队 | 项目全局视图、进度里程碑、风险识别 |
| **售前解决方案** | 管理投标文件、技术方案、客户沟通记录 | 报价单管理、方案文档分类、客户信息提取 |
| **开发工程师** | 管理技术方案、代码规范、接口文档、测试报告 | 技术文档分类、代码规范管理 |
| **客户成功经理** | 管理验收文档、培训资料、交接清单 | 验收进度、签字状态、交接完整性 |

## 三、功能需求详述

### F01 项目管理

#### F01.1 创建项目

**业务场景**：用户开始管理一个新客户项目。

**需求描述**：
1. 用户输入项目名称（必填）
2. 选择文件分类方式：
   - **按阶段分类**：预定义 11 个文件分类阶段，文件自动归类到对应阶段文件夹
   - **按内容分类**：AI 自动识别内容，动态创建分类文件夹
3. 项目创建后：
   - 在用户指定的存储位置（默认 `{userData}/projects`，可自定义）创建项目文件夹
   - 自动创建 11 个阶段子文件夹（按阶段分类时）
   - 自动创建 `.ai/` 元数据目录
   - **项目阶段默认为「售前」**
4. 文件系统创建失败时回滚数据库记录

**验收标准**：
- [ ] 输入名称后能成功创建项目
- [ ] 文件夹在正确位置创建
- [ ] 阶段子文件夹完整（11个）
- [ ] 项目阶段默认为「售前」
- [ ] 创建失败时数据库不残留孤儿记录

#### F01.2 项目列表

**业务场景**：用户管理多个项目，需要快速找到目标项目。

**需求描述**：
1. 支持**卡片视图**和**表格视图**切换
2. 卡片视图：
   - 每张卡片包含：顶部彩色条（4px，颜色跟随项目状态）、项目名称、状态标签、文件数量、分类方式标签、当前阶段、更新时间
   - 网格布局：`repeat(auto-fill, minmax(320px, 1fr))`，gap 20px
   - Hover 效果：阴影变化 + 微上移
   - Hover 时显示编辑/删除按钮
3. 表格视图：标准 AntD Table
4. 搜索：按项目名称模糊搜索
5. 筛选：按**项目阶段**（售前/进行中/关闭）
6. 排序：按更新时间、创建时间、名称

**验收标准**：
- [ ] 卡片视图布局正确，无遮挡
- [ ] 搜索实时过滤
- [ ] 项目阶段筛选有效
- [ ] 排序正确

#### F01.3 编辑项目

**业务场景**：用户需要修改项目信息。

**需求描述**：
1. 支持修改**项目阶段**（售前/进行中/关闭）
2. 支持修改项目名称
3. 支持修改分类方式（从阶段分类改为内容分类，或反之）
4. 修改后立即在界面反映

#### F01.4 删除项目

**业务场景**：用户需要删除不再管理的项目。

**需求描述**：
1. 二次确认弹窗
2. 级联删除：数据库记录 + 物理文件夹 + 对话记录
3. 删除后列表自动刷新

#### F01.5 打开文件夹

**业务场景**：用户需要在系统文件管理器中查看项目文件。

**需求描述**：一键打开项目在本地磁盘上的文件夹。

### F03 AI 智能分类

#### F03.1 单文件分类

**业务场景**：用户对单个文件触发 AI 分析。

**需求描述**：
1. 点击文件旁的"分类"按钮
2. AI 读取文件内容，判断：
   - **文件分类阶段**（11个阶段之一）
   - **项目阶段**（售前/进行中/关闭）— 用于判断是否需要推进
   - 文件内容摘要
   - 关键信息（项目编号、合同号等）
3. 分类完成后更新数据库 + 移动物理文件
4. 自动检测签字（PDF/图片）
5. **如果文件阶段高于当前项目阶段，弹窗询问是否推进项目阶段**

#### F03.2 一键分类

**业务场景**：用户有大量未分类文件，需要批量处理。

**需求描述**：
1. 点击"一键分类"按钮，对所有**分类阶段为「未分类」**的文件批量触发 AI 分类
2. 显示进度（当前/总数）
3. 支持取消
4. 分类完成后自动刷新

#### F03.3 分类 Prompt

**需求描述**：
1. 分类 Prompt 支持用户自定义（设置页面）
2. 默认 Prompt 包含 11 个**文件分类阶段**的描述
3. 保存后立即生效

## 四、Git工作流

### 4.1 分支策略

- `main`: 主分支，稳定版本
- `develop`: 开发分支
- `feature/*`: 功能分支
- `fix/*`: 修复分支

### 4.2 提交规范

使用语义化提交：

```
<type>(<scope>): <subject>

<body>

<footer>
```

**Type类型**：
- `feat`: 新功能
- `fix`: 修复bug
- `docs`: 文档更新
- `style`: 代码格式（不影响功能）
- `refactor`: 重构
- `test`: 测试相关
- `chore`: 构建/工具相关

**示例**：
```
feat(project): 添加自定义项目存储路径设置

- 设置页面新增存储路径配置项
- 支持浏览按钮选择路径
- 路径不存在时自动创建

Closes #123
```

### 4.3 协作流程

1. 从`develop`创建功能分支
2. 开发完成后提交PR
3. 代码审查通过后合并
4. 定期将`develop`合并到`main`
