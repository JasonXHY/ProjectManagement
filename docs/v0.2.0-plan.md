## 一、版本定位

v0.2.0 是 PMAer 的**业务深化版本**，核心目标：
1. **项目卡片信息增强** — 让项目经理一眼掌握项目全貌
2. **项目转交功能** — 支持团队协作，新成员快速了解项目
3. **Markdown预览优化** — 提升文档查看体验
4. **细节打磨** — 修复遗留问题，提升整体质量

## 二、功能模块详细设计

### 模块A：项目卡片信息增强

#### A0. 设计原则（Qoder审查要点）

1. **首页项目卡片**：保持简洁，不放里程碑进度（移到详情页）
2. **详情页卡片**：按阶段差异化展示，支持用户自定义
3. **所有新增UI**：必须经过Qoder根据UI设计原则审查
4. **卡片系统**：先发散设计，再收敛保留

#### A1. 首页项目卡片（简化版）

**当前首页卡片信息**：项目名称 + 状态标签 + 分类方式 + 更新时间 + 里程碑进度

**修改后首页卡片**：移除里程碑进度，保持简洁

```
┌─────────────────────────────────────────────┐
│ ████ 项目名称                          状态 │  ← 彩色条 + 标题 + 状态标签
│                                             │
│ 📁 12 文件 | ✅ 8 已分类 | ⚠️ 4 待处理     │  ← 文件统计行
│                                             │
│ ─────────────────────────────────────────── │
│ 3天前更新                    [管理] [编辑]   │  ← 底部操作栏
└─────────────────────────────────────────────┘
```

**说明**：首页卡片聚焦项目基本信息，里程碑进度等详情放到项目详情页的专门卡片中。

#### A2. 详情页卡片系统（核心设计）

**设计思路**：不同阶段展示不同侧重点的卡片，支持用户自定义。

**阶段差异化卡片**：

| 阶段 | 重点卡片 | 说明 |
|------|----------|------|
| 售前 | 报价单、竞品分析、客户需求 | 侧重商务信息 |
| 启动 | 项目计划、团队分工、启动会议 | 侧重启动准备 |
| 需求 | 需求文档、需求评审、变更记录 | 侧重需求管理 |
| 方案 | 技术方案、架构设计、选型报告 | 侧重技术决策 |
| 构建 | 开发进度、代码审查、构建记录 | 侧重开发过程 |
| 测试 | 测试用例、缺陷清单、测试报告 | 侧重质量保障 |
| 上线 | **问题清单**、部署文档、上线检查表 | 侧重上线风险 |
| 验收 | **签字文件列表**、验收报告、交付物清单 | 侧重验收流程 |
| 转客户成功 | 客户培训、运维手册、交接清单 | 侧重客户交付 |
| 关闭 | 项目总结、经验教训、归档清单 | 侧重项目复盘 |

**用户自定义**：
- 每个阶段页面右上角有 [更多卡片] 按钮
- 点击后弹出卡片选择面板，用户勾选要显示的卡片
- 选择结果保存到用户设置

#### A3. 卡片发散设计（MiMo + Qoder各自 brainstorm）

**MiMo 提出的卡片想法**（供参考，Qoder可增删改）：

| 卡片名称 | 展示内容 | 适用阶段 |
|----------|----------|----------|
| 文件统计 | 总数/已分类/待处理/各阶段分布 | 全部 |
| 里程碑时间轴 | 最近完成节点 + 下一个节点要求 | 全部 |
| 签字文件列表 | 待签/已签/已拒文件清单 | 验收 |
| 问题清单 | 关键问题列表 + 状态 | 上线/测试 |
| AI分析摘要 | 项目整体分析 + 关键发现 | 全部 |
| 关键客户信息 | 联系人/决策者/偏好 | 售前/启动 |
| 项目难点 | 技术难点 + 风险点 | 方案/构建 |
| 待办事项 | 待处理任务列表 | 全部 |
| 最近活动 | 最近上传/分类/分析的文件 | 全部 |
| 团队分工 | 成员 + 负责模块 | 启动/需求 |
| 合同信息 | 合同号/金额/付款节点 | 售前/验收 |
| 交付物清单 | 需要交付的文件列表 | 验收/关闭 |

**Qoder 需要补充的卡片**（请根据业务场景补充更多）：

- 请Qoder从项目经理、顾问、团队成员等不同角色视角思考
- 哪些信息是高频查看的？
- 哪些信息需要实时更新？
- 哪些信息需要预警提示？

#### A4. 里程碑展示（不使用进度比例）

**设计原则**：不显示进度百分比，因为里程碑难以量化为比例。

**展示方式**（Qoder设计）：

```
┌─────────────────────────────────────────────┐
│ 里程碑                                      │
├─────────────────────────────────────────────┤
│ ✓ 2026-06-15 需求确认完成                    │
│ ✓ 2026-06-18 方案评审通过                    │
│ → 下一步：2026-06-30 前提测                  │
│                                             │
│ ⏳ 2026-07-15 上线部署（待定）                │
│ ⏳ 2026-07-20 客户验收（待定）                │
└─────────────────────────────────────────────┘
```

**展示逻辑**：
- ✓ 已完成节点：显示完成日期
- → 下一个关键节点：高亮显示，附带要求说明
- ⏳ 未到节点：灰色显示，日期待定

**数据来源**：`projects.milestones` JSON 字段

**Qoder需要设计**：
- 节点之间的视觉连接方式
- 如何突出"下一步"节点
- 是否需要时间轴图形
- 如何处理节点延期情况

### 模块B：项目转交功能（核心新功能）

#### B1. 功能概述

**场景1：整体转交** — 项目经理将整个项目交给接手人
- 打包所有项目文件 + 元数据 + AI摘要
- 生成标准格式的转交包
- 接手人导入后可完整还原项目上下文

**场景2：协作转交** — 邀请同事参与项目特定部分
- AI根据描述智能选择相关文件
- 生成精简的转交包（仅包含必要信息）
- 接手人快速了解项目背景和负责模块

#### B2. 转交包格式设计

**格式选择**：JSON + 文件附件（非压缩包）

**理由**：
1. JSON是AI易读格式，云端模型可直接解析
2. 结构化数据便于导入和校验
3. 文件保持原始格式，无需解压

**转交包结构**：

```
pmaer-handover/
├── handover.json          ← 转交元数据（必须）
├── project-summary.md     ← 项目摘要（AI生成）
├── key-info.md           ← 关键信息（客户/合同/联系人）
├── files/                 ← 项目文件（按分类组织）
│   ├── 售前/
│   │   ├── 报价单.xlsx
│   │   └── 需求文档.docx
│   ├── 需求/
│   │   └── 需求规格说明书.docx
│   └── ...
└── context.md            ← 转交说明（发送方填写的描述）
```

**handover.json 结构**：

```json
{
  "format_version": "1.0",
  "type": "full" | "selective",
  "project": {
    "name": "项目名称",
    "category_type": "stage",
    "current_stage": "进行中",
    "created_at": "2026-06-01T10:00:00Z",
    "milestones": [...],
    "metadata": {...}
  },
  "files": [
    {
      "filename": "需求文档.docx",
      "category": "需求",
      "subcategory": "需求规格",
      "stage": "需求",
      "file_type": "application/docx",
      "file_size": 102400,
      "content_extracted": "（提取的文本内容）",
      "signature_status": "signed",
      "ai_summary": "（AI摘要）",
      "ai_key_info": "（关键信息）"
    }
  ],
  "summary": {
    "project_overview": "（AI生成的项目概览）",
    "key_clients": "（关键客户信息）",
    "challenges": "（项目难点）",
    "pending_items": "（待办事项）",
    "team_context": "（团队上下文）"
  },
  "handover_note": "（发送方填写的转交说明）",
  "exported_at": "2026-06-22T15:00:00Z",
  "exported_by": "PMAer v0.2.0"
}
```

#### B3. 转交流程设计

**整体转交互动流程**：

```
用户点击"转交项目"
    ↓
选择转交模式：[整体转交] [协作转交]
    ↓
┌─ 整体转交 ─┐    ┌─ 协作转交 ─┐
│ 直接打包    │    │ 填写描述    │
│ 所有文件    │    │ ↓          │
│            │    │ AI分析选择  │
│            │    │ ↓          │
│            │    │ 预览确认    │
└────┬───────┘    └────┬───────┘
     ↓                 ↓
生成转交包（文件夹）
     ↓
提供转交链接/文件路径
```

**协作转交的AI决策流程**：

```
用户输入："新同事负责前端开发，需要了解UI规范和接口文档"
    ↓
系统Prompt：你是项目转交助手，根据以下描述决定打包哪些文件...
    ↓
AI分析项目文件列表 + 用户描述
    ↓
返回：需要打包的文件列表 + 理由
    ↓
用户确认/调整
    ↓
生成精简转交包
```

**AI Prompt模板**：

```
你是PMAer项目转交助手。根据以下信息，决定应该打包哪些文件给新同事：

## 新同事描述
{user_description}

## 项目文件列表
{file_list_with_categories}

## 项目摘要
{project_summary}

请返回JSON格式：
{
  "selected_files": ["文件名1", "文件名2", ...],
  "reason": "选择这些文件的原因",
  "suggested_reading_order": ["文件名1", "文件名2", ...],
  "key_context": "新同事需要了解的关键上下文"
}
```

### 模块C：Markdown预览优化

#### C1. 问题现状

当前AI回复已支持Markdown渲染（react-markdown），但：
1. 项目中的.md文件打开仍是纯文本
2. 无语法高亮
3. 无表格渲染
4. 无图片嵌入支持

#### C2. 优化方案

**方案：统一Markdown渲染组件**

```typescript
// 新增组件：src/components/MarkdownPreview/
// - MarkdownPreview.tsx — 主组件
// - code-highlight.ts — 代码高亮配置

<MarkdownPreview 
  content={fileContent}
  fileName="readme.md"
  maxHeight={600}
/>
```

**技术选型**：
- 复用现有 `react-markdown` + `remark-gfm`
- 新增 `rehype-highlight` 用于代码语法高亮
- 新增 `rehype-raw` 支持HTML嵌入

**应用场景**：
1. AI回复渲染（已有，优化样式）
2. .md文件预览（新增）
3. 项目摘要预览（新增）
4. 转交包中的context.md预览（新增）

#### C3. 文件预览入口

在文件列表中，.md文件增加"预览"按钮：

```
📄 README.md    [预览] [打开] [分类]
```

点击"预览"弹出模态框显示渲染后的Markdown。

## 三、项目经理视角：卡片信息价值

### 项目经理关注什么？

作为项目经理，打开PMAer首页时，需要一眼回答以下问题：

| 问题 | 对应卡片信息 | 紧迫度 |
|------|-------------|--------|
| 项目整体进度如何？ | 里程碑时间轴（详情页） | 高 |
| 有哪些文件待处理？ | 待分类文件数 | 高 |
| 签字流程走到哪了？ | 签字文件列表（详情页） | 高 |
| 最近有什么新文件？ | 最近活动时间 | 中 |
| AI分析有什么发现？ | 关键问题数 | 中 |
| 团队最近在做什么？ | 文件分类分布 | 低 |

### 卡片信息优先级矩阵

```
          高紧迫度
              │
    ┌─────────┼─────────┐
    │ 待分类  │ 签字    │
    │ 文件数  │ 文件列表│
    │         │         │
高──┼─────────┼─────────┤──低
价  │ 最近    │ 关键    │ 价
值  │ 活动    │ 问题数  │ 值
    │         │         │
    └─────────┼─────────┘
              │
          低紧迫度
```

## 四、顾问视角：转交功能价值

### 新顾问加入项目时需要了解什么？

| 信息类别 | 具体内容 | 转交包提供方式 |
|----------|----------|---------------|
| **项目背景** | 项目目标、范围、时间线 | project-summary.md |
| **客户信息** | 关键联系人、决策者、偏好 | key-info.md + metadata |
| **团队情况** | 分工、接口人、协作方式 | context.md（手动填写） |
| **技术栈** | 使用的技术、架构、规范 | 文件内容提取 |
| **当前状态** | 进行中的工作、待办事项 | AI分析摘要 |
| **历史决策** | 为什么这样做、做过什么选择 | 文件内容 + AI总结 |
| **风险点** | 已知问题、潜在风险 | AI分析 + 手动补充 |
| **验收标准** | 客户验收要求、质量标准 | 需求文档 + 合同 |

### 协作转交的智能选择逻辑

AI应根据新同事的角色选择相关文件：

| 角色 | 优先选择的文件类型 |

## 六、风险与依赖

### 技术风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| AI文件选择不准确 | 协作转交给出无关文件 | 允许用户手动调整选择 |

## 九、总结

v0.2.0 是 PMAer 从"个人工具"向"团队协作工具"演进的关键版本。项目转交功能将打破信息孤岛，让项目知识在团队中流动。同时，卡片系统增强让项目经理能够更高效地掌控项目全局。

**核心价值主张**：
- **一眼掌握**：项目卡片让关键信息触手可及
- **知识传递**：转交功能让项目经验可传承
- **智能辅助**：AI帮助筛选最相关的转交内容
- **灵活定制**：用户自定义每个阶段展示哪些卡片
